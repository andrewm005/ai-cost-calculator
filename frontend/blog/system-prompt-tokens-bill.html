How System Prompt Tokens Show Up on Your Bill

The system prompt is the instruction block that ships on every call to the model. Persona, examples, tool definitions, retrieved context — every word is a token billed at the input rate. A 1,500-token system prompt on GPT-4o is a fixed $0.00375 per call. At one million calls per month, that prompt is a $3,750 line item on its own, before the user message, before the output, before the cache. This article is about that line item.

What a system prompt actually contains

A typical production system prompt has five sections. A persona block that tells the model what role to play (50-200 tokens). A tone block that adjusts style (50-100 tokens). A few-shot examples block that demonstrates the desired output shape (200-2,000 tokens). A tool definitions block that lists the tools the model can call (200-1,000 tokens). And a retrieved context block that injects the document chunks the model needs to answer (variable, often 1,000-10,000 tokens).

Each section grew incrementally. The persona started at 50 tokens and gained edge-case clauses until it was 200. The examples block started at two examples and grew to six. The tool definitions grew from one tool to seven. Six months into production, the system prompt is 3,500 tokens and nobody on the team remembers why each clause is there. This is the normal growth pattern, and it is the source of the hidden tax.

The per-call math

On GPT-4o at $2.50 per 1M input tokens, a 1,500-token system prompt costs $0.00375 per call. On Claude Sonnet 4 at $3.00 per 1M, the same prompt is $0.00450. On Gemini 2.5 Pro at $1.25 per 1M, the same prompt is $0.00188 — the cheapest flagship on input by a meaningful margin.

At volume, the per-call tax scales linearly. One million calls per month on GPT-4o is $3,750. Ten million calls is $37,500. One hundred million calls is $375,000. The system prompt alone, with no user message and no output, is a budget line.

Why this is invisible in the bill

Most LLM bills break down by model and date, not by prompt section. The system prompt tokens are mixed in with the user message and any retrieved context, billed at the input rate. A team looking at the bill sees "input tokens: 3,500 average per call, $0.00875 per call on GPT-4o." They do not see that 2,000 of those tokens are the system prompt and could be cached, trimmed, or moved to a different model.

The fix is instrumentation. Most production systems can instrument the per-call input token count by section: system prompt tokens, retrieved context tokens, user message tokens. A few hours of work surfaces the tax directly. A team that sees the system prompt is 60% of input tokens can act on it. A team that doesn't, doesn't.

Three ways to reduce the tax

Trim the prompt. The cheapest fix. Move examples out of the system prompt into one-shot user messages. Remove redundant instructions. Combine clauses that say the same thing twice. A reasonable goal: cut the system prompt by 30% without changing the model's behavior. For a 1,500-token prompt, that's 450 tokens saved, or $1,125/month at 1M calls on GPT-4o.

Cache the prompt. The biggest fix. Anthropic's cache rate is 10% of input. Claude Sonnet 4's input is $3.00, so the cache is $0.30. A 1,500-token system prompt on every call, cached, is $0.00045 per call vs $0.00450 without caching — a 10x reduction. At 1M calls per month, that's $4,050/month saved. OpenAI's cache rate is 50% off input on the GPT-4.1 family; GPT-4o does not have a published cache rate on OpenRouter as of mid-2026. The prompt caching article has the structural details.

Route the prefix. Move the system prompt out of the per-call API request entirely. Some production systems use Anthropic's prompt caching with a long TTL on the system prefix, then send only the user message and the retrieved context per call. The model still sees the system prompt (cached), but the application does not re-send it. This is mechanically the same as prompt caching but easier to reason about in a multi-tenant system where the prefix varies by customer.

When the system prompt needs to be long

Some workloads genuinely need a long system prompt. A code-review agent that ships the project's coding conventions plus the file tree plus the diff plus the prior review history might have a 5,000-token system prompt on every call. That is not waste — that is the agent's working memory.

The right framing for these workloads is to make sure the prompt is structured for cache hits. A 5,000-token prefix that hits the cache every call is dramatically cheaper than the same 5,000 tokens re-billed at the input rate. Anthropic's prompt caching supports up to four cache breakpoints per prompt, so a long system prompt can be split into the stable prefix (cacheable, hits every time) and the variable suffix (per-call, billed at input rate). The stable prefix is the bulk of the system prompt for most long-context workloads.

A concrete example

A customer support chatbot running on Claude Sonnet 4 ships a 2,000-token system prompt (persona + tone + 8 few-shot examples + 3 tool definitions) on every call. The user message averages 200 tokens. The assistant's reply averages 300 tokens. At 100,000 calls per month:

The audit

A 30-minute audit reveals the tax. Step one: log the system prompt token count on a representative sample of production calls. Step two: multiply by the input rate and the monthly volume. Step three: compare to the line items on the bill. Step four: if the system prompt is more than 40% of input tokens, treat it as a structural problem and ship a fix.

The fix is usually cache + trim, in that order. Caching buys the 10x reduction with no behavior change. Trimming buys the 30% reduction with a behavior risk that has to be evaluated. Most teams find that 70% of the system prompt is genuinely necessary, 20% is redundant and trimmable, and 10% is actively harmful (instructions that confuse the model or fight the model's defaults).

Frequently asked questions

What is a system prompt?

The system prompt is the instruction block the application sends to the model on every API call. It usually contains persona instructions, examples, tool definitions, retrieved context, and any other persistent guidance the model needs to do its job. It is sent as part of the input and is billed at the input token rate.

How much does a typical system prompt cost?

On GPT-4o at $2.50 per 1M input tokens, a 1,500-token system prompt costs $0.00375 per call. At 1M calls per month, that system prompt alone is $3,750/month. On Claude Sonnet 4 at $3.00 per 1M, the same prompt is $4,500/month. With prompt caching on Claude, it drops to $450/month.

Do all system prompts get cached?

No. Caching is prefix-based: the model has to see the same opening tokens to count them as cached. If the system prompt comes before the user message in the prompt array, the system portion is cacheable. If the user message comes first and the system prompt is appended at the end, the cache misses every call.

What's the right size for a system prompt?

For short-form workloads (chat, classification, extraction), keep the system prompt under 800 tokens. For long-context workloads (RAG, multi-file code review), under 1,500 tokens. Beyond 2,000 tokens, there is usually a 30% trim available without losing capability.
