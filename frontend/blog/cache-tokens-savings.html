Prompt Caching: How Anthropic and Gemini Discount Repeat Tokens

Most teams are paying full price for the same system prompt on every API call when they could be paying 10% of the price. The cache is sitting right there, in the same request, on every vendor. Anthropic publishes it explicitly. OpenAI publishes it at 50% off. Google's Gemini has implicit caching that activates on prompts above a token threshold. A team that ships a 2,000-token system prompt on every call without turning the cache on is leaving 78% of that input line item on the table, every month, for the rest of the product's life. This article is about why it stays off, and what it costs.

Concrete example: a 2,000-token system prompt on Claude Sonnet 4

Take a representative production application: 2,000-token system prompt, 300-token user message, 250-token assistant reply. One million calls per month on Claude Sonnet 4, input $3.00 per 1M tokens, output $15.00 per 1M:

The 90% reduction is the upper bound; in practice the user message changes every call, so the cache hits the system prompt only. The 78% reduction is realistic for any application with a stable system prompt shipped every call. On a chat-shaped workload with a roughly 50/50 input-output cost split, that 78% reduction on the input line works out to a 50% reduction on the total bill. The GPT-4o vs Claude Sonnet 4 comparison applies the same math to the flagship pair.

How the cache rate works

The cache is a vendor-side optimization. The vendor stores the opening tokens of recent requests in memory. When a new request comes in, the vendor checks whether the opening tokens match a recent prefix. If they do, the vendor serves the cached prefix from memory and bills those tokens at the cache rate instead of the input rate. The model produces the same output as if the prefix had been processed fresh. The cache is invisible to the model and to the application's user.

Anthropic's cache rate is 10% of input. Claude Sonnet 4 lists $3.00 per 1M input and $0.30 per 1M cache, a 10x discount. Claude Opus 4 is $15.00 input / $1.50 cache. Claude Haiku 4.5 is $1.00 input / $0.10 cache. The 10x ratio holds across the family. OpenAI's cache rate is 50% off input on supported models. GPT-4.1 is $2.00 input / $0.50 cache. GPT-4.1 mini is $0.40 / $0.10. GPT-4o mini is $0.15 / $0.075. GPT-4o itself does not publish a cache rate in mid-2026; the cache is available on the GPT-4.1 family and on GPT-4o mini, but not on the flagship GPT-4o. Google's Gemini family uses implicit caching. Gemini 2.5 Pro caches at $0.125 per 1M, which is 10% of the $1.25 input. Gemini 2.5 Flash caches at $0.030 per 1M, also 10% of input. The implicit cache activates above a token threshold (typically 1,000 to 2,000 tokens) and requires no explicit breakpoint.

When the cache hits and when it misses

The cache is prefix-based and the prefix has to be byte-identical across calls. The opening tokens of the new request have to match the opening tokens of a recent request, exactly. A single variable token at the start of the system prompt — a timestamp, a request ID, a per-tenant variable — invalidates the entire cache on every call. The structure of the prompt, not its content, is the difference between a 90% cache hit rate and a 0% cache hit rate. A prompt with the user message first and the system prompt appended at the end is a cache miss every call.

The cache TTL matters less than most teams expect. Anthropic's default TTL is 5 minutes, extendable to 1 hour with an explicit breakpoint on the prompt. OpenAI's TTL is 5 to 10 minutes depending on the model. Google's implicit cache has no explicit TTL. A high-traffic application (more than one call per minute on the same prefix) hits the cache most of the time regardless. A low-traffic application (less than one call per five minutes on the same prefix) misses often and the savings shrink. For a low-traffic application, the right move is to set a long-TTL breakpoint on the stable prefix and re-send the prefix periodically to keep the cache warm. The cost of keeping a 5,000-token prefix warm once per hour is the cache rate for 5,000 tokens every hour — effectively free.

The implementation checklist

Five things have to be true for prompt caching to deliver the savings. First, the cacheable prefix is at the start of the prompt, not appended at the end. Second, the prefix is byte-identical across calls — no timestamps, no request IDs, no per-call variables in the cached portion. Third, breakpoint markers are set explicitly for the Anthropic API, or the implicit cache is large enough to activate for Google. Fourth, the cache TTL is longer than the typical interval between calls with the same prefix. Fifth, the application sends the prefix on every call even when the user message is short; the vendor needs the prefix to do the cache lookup.

Items one through three are mechanical and ship in an afternoon. Items four and five require some thought about traffic patterns and prompt structure but are straightforward. The savings compound: a 78% reduction on the input line is roughly a 50% reduction on the total bill for a chat-shaped workload. A RAG workload with a 4,000-token retrieved context that's the same across calls sees the same 78% reduction on the cached portion. A coding agent with a 1,500-token system prompt cached across every loop of an agentic task sees the same effect compounded across ten loops. Clearly, the cache is the largest single line item most teams are not collecting.

What doesn't cache

The user message cannot be cached. Per-call retrieved context cannot be cached. The output is not cacheable. Reasoning tokens are not cacheable. Tool call results are not cacheable — they change every call. For an application with a 500-token system prompt and a 5,000-token user message, the cacheable portion is 500 tokens out of 5,500, roughly 9% of input. The cache discount on 9% of input is a 9% reduction on the input line — meaningful, but not the 78-90% reduction the same application gets with a 2,000-token system prompt. The cache is most valuable when the cacheable prefix is a large fraction of the total input.

Frequently asked questions

How much does prompt caching actually save?

Anthropic's cache rate is 10% of input. OpenAI's cache rate is 50% off input on supported models. Google's Gemini family caches implicitly at 10% of input above a token threshold. For a 2,000-token cacheable system prompt shipped every call, the cache typically cuts the input line by 50% to 90%.

Does the cache have a time-to-live?

Yes. Anthropic's TTL is 5 minutes by default, extendable to 1 hour with an explicit breakpoint. OpenAI's is 5 to 10 minutes depending on the model. Google's implicit cache has no explicit TTL — it activates when the prefix matches a recent request.

What breaks the cache?

Any change to the opening tokens. Anthropic and OpenAI both require byte-identical prefixes to count as a cache hit. A single variable token in the cached portion invalidates the entire prefix from that token forward.

How should I structure the prompt for cache hits?

Put the cacheable content first. System prompt, tool definitions, and stable retrieved context belong at the start; the user message and per-call context go at the end. Anthropic supports up to four cache breakpoints per prompt.
