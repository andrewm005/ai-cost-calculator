Prompt Caching: How Anthropic and Gemini Discount Repeat Tokens

Prompt caching is the largest single cost saver in any LLM bill that ships a long prefix on every call. Anthropic's cache rate is 10% of input. OpenAI's cache rate is 50% off input on supported models. Google's Gemini family has implicit caching on prompts over a token threshold. Most production applications are eligible for cache hits on the bulk of their input tokens and never turn it on. This article is about how to turn it on and what it saves.

What prompt caching is

Prompt caching is a vendor-side optimization. When you send a prompt to the model, the vendor checks whether the opening tokens match a recent request. If they do, the vendor serves the cached prefix from memory and charges you the cache rate for those tokens instead of the input rate. The model still produces the same output as if it had processed the prefix fresh — the cache is invisible to you and to the model.

The catch is that the cache is prefix-based. The vendor has to see the same opening tokens to count them as cached. If the user message comes first and the system prompt is appended at the end, the cache misses every time. The structure of your prompt is the difference between a 90% cache hit rate and a 0% cache hit rate.

The cache rates

Anthropic's cache rate is 10% of input. Claude Sonnet 4 lists input at $3.00 per 1M tokens and cache at $0.30 per 1M. Claude Opus 4 is $15.00 input / $1.50 cache. Claude Haiku 4.5 is $1.00 input / $0.10 cache. A 10x discount across the board.

OpenAI's cache rate is 50% off input on supported models. GPT-4.1 is $2.00 input / $0.50 cache. GPT-4.1 mini is $0.40 / $0.10. GPT-4o mini is $0.15 / $0.075. GPT-4o does not publish a cache rate on OpenRouter as of mid-2026 — the cache is supported on the GPT-4.1 family and on GPT-4o mini but not on the base GPT-4o.

Google's Gemini family uses implicit caching. Gemini 2.5 Pro caches at $0.125 per 1M (10% of the $1.25 input rate). Gemini 2.5 Flash caches at $0.030 per 1M (10% of the $0.30 input rate). The implicit cache activates when the prefix matches a recent request above a token threshold — typically 1,000-2,000 tokens. No explicit breakpoint is required.

The cache TTL

Caches expire. Anthropic's default TTL is 5 minutes; it can be extended to 1 hour with an explicit cache breakpoint on the prompt. OpenAI's cache TTL is 5-10 minutes depending on the model. Google's implicit cache has no explicit TTL — it activates when the prefix matches a recent request.

The TTL matters less than most teams think. A high-traffic application (more than 1 call per minute on the same prefix) hits the cache most of the time. A low-traffic application (less than 1 call per 5 minutes on the same prefix) misses often and the cache savings are smaller.

For a low-traffic application, the right move is to set an explicit long-TTL breakpoint on the stable prefix and re-send the prefix periodically to keep the cache warm. The cost of keeping a 5,000-token prefix warm once per hour is the cache rate for 5,000 tokens every hour — effectively free.

How to structure prompts for cache hits

The mechanical rule: put the cacheable content first. The system prompt, tool definitions, and any stable retrieved context go at the start of the prompt. The user message and any per-call retrieved context go at the end.

Anthropic supports up to four cache breakpoints per prompt. A long system prompt can be split into the stable prefix (cacheable, hits every time) and the variable suffix (per-call, billed at input rate). The breakpoint is a marker that tells the vendor "everything before this is cacheable, everything after is fresh." A 5,000-token system prompt with a breakpoint after the first 4,000 tokens means the first 4,000 hit the cache and the last 1,000 bill at input rate.

A common structural mistake is to put a timestamp or a request ID at the start of the system prompt. That single variable token invalidates the entire cache on every call. Timestamps and request IDs go at the end of the prompt or inside the user message.

What does not cache

The cache is for the prefix. The user message and the per-call retrieved context are not cacheable — they change every call. The output is not cacheable. Reasoning tokens are not cacheable. Tool call results are not cacheable (they change every call).

For an application with a 500-token system prompt and a 5,000-token user message, the cacheable portion is 500 tokens out of 5,500 — about 9% of the input. The cache discount on 9% of input is a 9% reduction on the input line. That's still meaningful but it's not the 78-90% reduction the same application gets with a 2,000-token system prompt. The cache is most valuable when the cacheable prefix is a large fraction of the total input.

The implementation checklist

Five things that have to be true for prompt caching to deliver the savings. First, the cacheable prefix is at the start of the prompt, not appended at the end. Second, the prefix is byte-identical across calls — no timestamps, no request IDs, no per-call variables in the cached portion. Third, the breakpoint markers are set explicitly for the Anthropic API (or the implicit cache is large enough to activate for Google). Fourth, the cache TTL is longer than the typical interval between calls with the same prefix. Fifth, the application sends the prefix on every call even when the user message is short — the vendor needs the prefix to do the cache lookup.

For most production applications, items 1-3 are mechanical and ship in an afternoon. Items 4-5 require some thought about traffic patterns and prompt structure but are also straightforward. The savings compound: a 78% reduction on the input line is roughly a 50% reduction on the total bill for a chat-shaped workload.

Frequently asked questions

How much does prompt caching save?

Anthropic's cache rate is 10% of input. OpenAI's cache rate is 50% off input on supported models. Google's Gemini family has implicit caching. For a workload with a 2,000-token cacheable system prompt shipped on every call, prompt caching typically cuts the input line by 50-90%.

Does the cache have a TTL?

Yes. Anthropic's cache TTL is 5 minutes by default, extendable to 1 hour. OpenAI's is 5-10 minutes. Google's implicit cache is best-effort with no explicit TTL. A high-traffic application hits the cache most of the time; a low-traffic application misses often.

What if the prefix changes between calls?

The cache misses. Anthropic and OpenAI both require the prefix to be byte-identical to count as a cache hit. If you change a single token, the cache miss starts from that token forward. Put stable content at the start; variable content at the end.

How do I structure my prompt for cache hits?

Put the cacheable content first — system prompt, tool definitions, stable retrieved context. The user message and per-call context go at the end. Anthropic supports up to four cache breakpoints per prompt. Timestamps and request IDs must not appear in the cached portion.
